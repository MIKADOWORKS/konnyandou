# 出力テンプレート集

タスク種別に応じて、以下のテンプレートを使って報告すること。
テンプレートの項目は省略可能だが、構造は維持する。

---

## A. コード変更の報告

```
## 結果: [成功 / 失敗 / 一部完了]

### 変更サマリー
[1-2行で何をしたか]

### 変更ファイル
| ファイル | 変更内容 |
|---------|---------|
| `path/to/file.ts` | 〇〇を追加 |
| ... | ... |

### ビルド: [成功 / 失敗（エラー内容）]

### 注意点（あれば）
- [他チームへの影響、残課題など]
```

## B. 調査・分析の報告

```
## 結論
[1-3行で調査結果の要約]

### 詳細

#### [調査項目1]
- [箇条書きで発見事項]

#### [調査項目2]
- [箇条書きで発見事項]

### 推奨アクション
| 優先度 | アクション | 理由 |
|--------|-----------|------|
| P0 | 〇〇 | △△のため |
| P1 | 〇〇 | △△のため |
```

## C. 設計仕様の報告

```
## 概要
[この設計が解決する課題を1-2行で]

### 設計内容

#### [セクション1]
[具体的な仕様]

#### [セクション2]
[具体的な仕様]

### 実装に必要なもの
| 項目 | 担当 | 工数目安 |
|------|------|---------|
| 〇〇 | エンジニア | X日 |

### 未決定事項（あれば）
- [ ] [ディレクター/オーナーの判断が必要な項目]
```

## D. バグ修正の報告

```
## 結果: [修正完了 / 暫定対処 / 未解決]

### 原因
[1-2行で根本原因]

### 修正内容
| ファイル | 修正内容 |
|---------|---------|
| `path/to/file.ts` | 〇〇を修正 |

### 再発防止（あれば）
- [バリデーション追加、テスト追加など]
```

## E. 新機能の提案

```
## 提案: [機能名]

### 背景
[なぜこの機能が必要か、1-2行]

### 仕様概要
- [主要な機能ポイントを箇条書き]

### 技術アプローチ
- [実装方針を箇条書き]

### 工数・インパクト
| 項目 | 値 |
|------|-----|
| 工数 | X日 |
| インパクト | [高/中/低] |
| リスク | [高/中/低] + 理由 |

### 依存関係
- [他タスク・他ロールへの依存]
```

## F. Sprint完了の報告（ディレクター用）

```
## Sprint X 完了報告

### 完了タスク
| # | タスク | 担当 | 状態 |
|---|--------|------|------|
| 1 | 〇〇 | エンジニア | 完了 |
| 2 | △△ | プロトEng | 完了 |

### 未完了（あれば）
| タスク | 理由 | 対応方針 |
|--------|------|---------|
| 〇〇 | ブロッカー: △△ | 次Sprintへ繰り越し |

### 次Sprintの推奨タスク
1. [最優先タスク]
2. [次点タスク]
```
